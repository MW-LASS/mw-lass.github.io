Designing URIs

Basic Construction of a URI

The following identifies the RECOMMENDED elements within a URI and forms the basic pattern:

http://{domain}/{type}/{concept} /{reference}

  • {Domain}: The preferable domain to use would be 'data.govt.nz', which is currently under review. In the meantime, the Manawatu-Wanganui domain and sub-domain is: data.mw.org.nz. OR Council's use their own domain and sub-domain for data (e.g., data.rangitikei.govt.nz).
  • {type}: Categories of resources: Real-world 'Thing'= id; document or data = doc; and definition = def. See next section 'Types of URIs' for full descriptions.
  • {concept}: A word or string to capture the essence of the real-world 'Thing' that the set names. For example, environment. A concept MUST be: lowercase, words separated by hyphens and singular (i.e., not 'environments'). For example, data.mw.org.nz/cemetery
  • {reference}: A specifc item, term or concept.

Types of URIs

The following table describes each type of URI with associated resource: Also refer to: Australia Guidelines...

Type of Resource Type of URI Definition
A real-world 'Thing'  (Non-Information Resource) Identifier URI (/id) Identifies some physical or abstract real-world 'Things'. Example of a real-world 'Thing': a person, a road, a lake. Example of abstract real-world 'Thing': a government, a council.
---
A regular document  (Information Resource) Document URI (/doc) Identifies a document. The Document URI should resolve directly to a document on the Web.
---
A 'Real-World Thing'  (Information Resource) Ontology URI (/def) Definitions of concepts and relations contained within the ontologies that define characteristics of a real- world 'Thing'. Example: a person class with properties such as 'firstName', 'lastName' etc. The ontology URI with return a definition or set of definitions about the real-world 'Thing'. That is the person class will return the properties listed above.

Hash (#) versus Slash (/) URIs

There are two types of URIs that depict resources which will require differnt approaches to handling. Slash URIs allows direct access to resources, whereas Hash URIs are easier to deploy. For more descriptive information, refer to: Australia Guide
